> 作者：peihanyu（腾讯程序员）

## 核心问题

「规则越多，AI 遵守得越差。把 200 条规则一次性灌进去，效果就像让一个实习生第一天上班时，你把公司全部规章制度从头到尾读一遍然后说"好了你去干活吧"。」

两个工程痛点：

1. **Token 是钱，试错是浪费** — Agent 在模糊需求前反复揣摩、多轮尝试，每一步都在烧 Token
2. **路径不确定，每次执行像开盲盒** — 同一个提示词，换个模型、换个上下文顺序，输出完全不同

> 「LLM 的注意力更像人类读一本很厚的操作手册——你知道所有内容都在里面，但真正能"活跃"在大脑中的只有你刚读过的那几页。」

## 方法

<table>
<tr><th>维度</th><th>内容</th></tr>

<tr><td><strong>核心理念</strong></td>
<td>

目标：**把 Agent 当成一个算法来用**——给它输入，它给你指定格式的输出。中间推理过程你不关心，但结果是确定的、可预期的。

> 「不改变河的本性，但给它修好渠。Agent 保留它最擅长的"理解人话、做判断、组织表达"的能力，但把所有确定性的事——流程顺序、数据格式、API 调用、状态管理——全部从它脑子里拿出来，交给一个它触碰不到的确定性程序。」

定位切换：

| | 旧角色 | 新角色 |
|---|---|---|
| Agent | 什么都要自己想的自由意志体 | 嵌入精密轨道里的决策引擎 |
| 输入 | 自然语言 | 自然语言 |
| 输出 | 自由文本 | 结构化 JSON 参数 |

</td></tr>

<tr><td><strong>分清 AI 擅长和不擅长的事</strong></td>
<td>

> 「凡是涉及精确格式、固定流程的事，AI 不靠谱；凡是涉及理解、判断、表达的事，AI 很在行。」

| AI 做得好的 | AI 做不好的 |
|---|---|
| 理解用户说"帮我创建项目" | 拼 HTTP 请求时漏 header |
| 从三个选项里选最合适的 | 写 YAML 时缩进错了 |
| 用友好语气回复"已创建" | 把 `userId` 写成 `user_id` |

</td></tr>

<tr><td><strong>CLI 接管一切确定性的事</strong></td>
<td>

引入一个 CLI 程序作为 Agent 的执行层：

| 角色 | Agent（大脑）| CLI（手脚）|
|---|---|---|
| 做什么 | 理解意图、收集参数、组织回复 | 调 API、写文件、管状态 |
| 沟通方式 | 只输出 JSON 参数 | 只返回 JSON 结果 |
| 不确定性 | 有（被限定在决策范围内）| 无（相同输入 = 相同输出）|

**旧方式**：Agent 自己拼 HTTP 请求 → 漏 header / 字段名拼错
**新方式**：Agent 只说 `调用 create_project，参数：name=foo` → CLI 负责剩下一切

> 「Agent 的不确定性，被 CLI 的确定性包裹住了。」

</td></tr>

<tr><td><strong>工具管理三层分离</strong></td>
<td>

当有 50 个工具时，全部 schema 灌进去上下文会爆炸。方案：

| 层级 | 内容 | Agent 何时可见 |
|---|---|---|
| 索引层 | 工具名 + 一句话描述 | 始终（一张表格）|
| 元数据层 | 字段名、类型、必填、枚举值 | 请求某工具时按需加载 |
| 规则层 | IGNORE/NOTE/ENUM 注解 | 融合在元数据输出中，Agent 不直接看到 |

> 「Agent 面对 50 个工具时，它的上下文里只多了一张 50 行的表格。而不是 50 份 schema 文档。」

关键设计：**Agent 永远不需要调用 read_file 去读工具说明**。CLI 是"看门人"——什么时候给什么信息，由 CLI 决定，不由 Agent 决定。Agent 的 read_file 能力被结构性地收走，犯错可能性被结构性地消灭。

discover 机制：每次激活时自动同步后端工具列表→增量对比→为新工具自动生成 .meta 参数文件和 .rules 规则（自动识别系统字段生成 IGNORE、提取枚举值）。**工具列表不是静态的，是每次激活时实时同步的。**

</td></tr>

<tr><td><strong>Workflow：把散落工具串成流水线</strong></td>
<td>

用户的真实需求不是"调一个 API"而是"搭一套权限体系"——需要按正确顺序调多个 API。

**Workflow 的核心机制（五个）**：

**1. 步进式披露** — Agent 永远只看到当前一步
流程是文件系统里的一组步骤文件。Agent 通过 `--start` / `--advance` 与流程交互，在任意时刻看到的永远只有当前这一步。改流程 = 改文件名前缀。

> 「AI 从来不需要"自由"。它需要的是"当前该做什么"的极度清晰。」

**2. Gate（门禁）** — 把开放题变成填空题
每个步骤文件的 YAML front matter 声明 gate schema，精确到字段名、类型、是否必填。`required` 字段没给→拒绝推进。schema 没声明的字段→丢弃。

> 「约束越强，AI 反而越自由。Gate 的作用是把开放题变成填空题。」

**3. 状态持久化** — 记忆不在 AI 脑子里
CLI 把流程运行状态写到磁盘 `.state` 文件。任何会话、任何模型调 `--current` 就能从断点续上。每步输入输出都记录，可审计。

**4. 模板变量** — 数据流由 CLI 管理
`{{gate.collect-project-info.project_name}}` 在 automated 步骤的 input_mapping 里引用前序字段。CLI 从 state 文件取值、渲染参数、执行工具——全程不经过 Agent。

**5. 三种步骤类型协奏** — interactive（Agent 干活）/ automated（CLI 自动执行）/ notification（Agent 展示）。Agent 提交一次数据，CLI 可能连续做多步才把控制权还回来。

**两个关键亮点**：
- Workflow 是"长出来"的，不是"写出来"的——在 `workflows/` 下新建文件夹就扩展了能力，不改代码
- Workflow 与 Skill 同构——可以无缝互转，好的 Skill 可以直接塞进另一个 Skill 当 Workflow 复用

</td></tr>

<tr><td><strong>自举：用 Skill 创造 Skill</strong></td>
<td>

> 「新建 Workflow 本身就是一个"多步骤、固定格式、确定性"的流程。」

做了第二个 Skill（workflow-creator）——Agent 分析需求→拆步骤→确定类型→设计 gate schema→规划数据流；配套 CLI 做文件生成（init / add-step / validate）。**系统不只能运行，还能自己长出新的能力。**

</td></tr>

</table>

## 关键引用

> 「问题不是"怎么写更好的提示词"。而是：怎么设计一个让 AI 在每个时刻都只需要关注最少信息的执行环境？」

> 「判断拐点：当你发现自己在提示词里反复写"注意不要……""确保先……""必须在……之前……"——你正在用自然语言模拟一个状态机。这就是该工程化的信号。」

> 「Skill 工程化设计的全部哲学：不是驯服 AI，而是为它构建一个它天然适合流淌的河道。」
