## 核心问题

「Agent 要做上下文压缩，如今几乎成为所有 Agent 必做的一环。有意思的是怎么做——六家产品，六种哲学。说明这件事没有显而易见的最优解。」

第一代方案的根本问题：「把压缩当突发事件处理，而不是一种持续维护的能力。」悬崖式触发——不溢出不动，一溢出就全量压缩。全量摘要丢细节（变量名、函数签名、错误堆栈），Token 估算粗糙（中英混合误差 30-50%），不区分信息价值，用户内容被一刀切。

| 产品 | 核心策略 | 一句话 |
|------|---------|--------|
| Claude Code | 五段流水线，按成本递增排列 | 便宜的本地操作先上，LLM 摘要兜底 |
| Codex CLI | 保留近期用户消息原文，其余→handoff 摘要 | 用户说的最准确，模型说的可重写 |
| OpenCode | 时间戳标记隐藏 + 结构化摘要 + 回放 | 不真删，理论上可恢复 |
| Cline | `/compact` 生成摘要后同任务内接续 | 自动+手动双模式 |
| Cursor | 自动摘要 + 提示开新对话 + 历史可搜索 | 压缩后仍能回溯 |
| Amp | 不做递归压缩，`/handoff` 开新线程 | 换线程比压缩好 |
| MemGPT/Letta | 上下文=RAM，历史=磁盘，Agent 自主换入换出 | 操作系统级内存调度 |

</td></tr>

<tr><td><strong>已接近共识的原则</strong></td>
<td>

1. **分层压缩**：不同内容用不同粒度——用户消息保留原文，工具输出可以摘要
2. **渐进式**：不是 0 或 1，而是按水位线逐步处理
3. **区分信息价值**：代码/命令结果/诊断日志 vs 普通聊天，压缩策略不同
4. **用户内容优先保护**：用户输入的代码代表意图，不能一刀切

</td></tr>

<tr><td><strong>四级水位线方案（MUR AI）</strong></td>
<td>

云端多用户场景下的落地方案：

- **水位 1（安全区）**：不做任何压缩
- **水位 2（预警区）**：自动摘要系统消息和长工具输出，用户消息保留原文
- **水位 3（压缩区）**：结构化摘要所有历史，保留当前任务相关的关键片段（引用块模式）
- **水位 4（上限区）**：强制截断最早的消息，触发补偿策略（写文件保存被截内容）

云端多用户额外考虑：多租户 Token 计费隔离、压缩不跨用户泄漏信息、摘要缓存在用户维度复用。

</td></tr>
