> 作者：jackjchou（腾讯程序员）

## 核心问题

「裸着的 AI 就像一个刚入职的新人，啥都得问；装了 Skill 之后，就像拿到了老员工整理的操作手册，照着就能干。」

典型痛点：知识太散（藏在 TAPD/Wiki/人脑里）、重复搬砖、做出来的东西不统一、新人上手慢、人走知识也走。

## 方法

<table>
<tr><th>维度</th><th>内容</th></tr>

<tr><td><strong>Skill 的本质</strong></td>
<td>

> 「Skill 就是给 AI 编程助手"加装"的能力包。本质上，它是一种结构化的 Prompt Engineering——通过标准的文件格式，把分散在人脑中的领域知识、操作流程和最佳实践，转化为 AI 可理解、可执行的指令集。」

物理上是一个文件夹：`SKILL.md`（核心指令）+ `scripts/`（辅助脚本）+ `references/`（参考文档）。核心三要素：**指令**（告诉 AI 怎么做）、**上下文**（补课项目背景）、**工具**（可直接用的脚本模板）。

</td></tr>

<tr><td><strong>渐进式加载机制</strong></td>
<td>

三级加载，按需消费 Token：

| 层级 | 加载时机 | 内容 | Token 参考 |
|------|--------|------|-----------|
| Level 1 | 常驻 | name + description（≤100 字）| ~50-150/个 |
| Level 2 | 匹配触发 | SKILL.md 正文（≤500 行）| ~2,000-5,000 |
| Level 3 | 按需引用 | 脚本、参考文档 | 按实际大小 |

> 「核心原则：Level 1 越精准越好（决定触发时机），Level 2 越精简越好（减少 Token 消耗），Level 3 放心放（按需加载不占常驻空间）。」

Skill 不是免费的——装 20 个 Skill，光 Level 1 就要吃掉 1,000-3,000 Token。

</td></tr>

<tr><td><strong>Description：成败的关键</strong></td>
<td>

> 「AI 就是靠 description 来判断"用户现在说的这个事，该不该用这个 Skill"。写得太笼统，AI 不知道啥时候该用；写得太窄，很多该触发的场景又漏了。」

用通用语言描述功能，辅以具体技术关键词，别用内部黑话：

```
# ❌ description: 处理 TCC 的 v3 迁移，适配 QCMS 的新规范
# ✅ description: 将项目中的旧版事务补偿组件迁移到 v3 版本。
#    适用于 Go 项目中使用 TCC（Try-Confirm-Cancel）模式，
#    需要对接新版内容管理系统 API 的场景。
```

**实用技巧**：自己想 20 个问题（一半该触发、一半不该触发），测试 AI 命中率，不够就调 description。

</td></tr>

<tr><td><strong>五个关键技巧</strong></td>
<td>

**1. 开头说清三件事**：做什么、为什么、要不要做。给出前置检查命令告诉 AI 什么时候跳过：

> 「把起点和终点说清楚——从哪迁到哪，别含糊。告诉 AI 什么时候不用做——给个前置检查条件。」

**2. 用祈使句下指令**，别用商量口吻。讲清为什么比列一堆 MUST 更有效：

> 「AI 要是理解了背后的道理，遇到你没想到的情况也能做出合理判断。光靠"MUST"只是死记硬背。」

**3. Before/After 对比**：三种方式——注释标注（简单变更）、完整文件对比（复杂变更）、Diff 格式（推荐）。

**4. Few-Shot 示例**：放 3-5 个高质量输入/输出示例，覆盖最常见、稍有变化、边界/特殊三种场景。

**5. 决策树与流程图**：ASCII 图让 AI 比读长段文字理解得更准确。

</td></tr>

<tr><td><strong>模块化拆分</strong></td>
<td>

拆分信号：超过 500 行、多个独立工作流、不同部分改动频率差异大。

结构：主 Skill（流程总览与编排）+ 子 Skill（各自独立可单独调用）。

> 「一个子 Skill 只管一件事（单一职责）。每个子 Skill 都能单独使用。」

关键步骤之间插入检查点命令——某步出错立即阻断，别一口气跑到底。

</td></tr>

<tr><td><strong>常见反模式</strong></td>
<td>

1. **大杂烩 Skill**：一个 Skill 塞三四件事，AI 不知道该用哪部分
2. **Description 用内部黑话**：AI 不知道 TCC/QCMS 是什么
3. **只有指令没有示例**：全是文字描述，无 Before/After
4. **步骤间无验证点**：一口气做完才检查，中间出错全白干
5. **写死具体数值**而非提供判断规则：换项目就失效

</td></tr>

</table>

## 关键引用

> 「Skill 和 Rule 的区别：Rule 是"底线"（始终生效），Skill 是"技能"（按需触发）。所有对话都该遵守的写 Rule，特定任务才需要的写 Skill。」

> 「与其一堆"MUST"，不如讲清楚为什么。AI 要是理解了背后的道理，遇到你没想到的情况也能做出合理判断。」
